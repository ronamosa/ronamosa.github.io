## Project Overview

Docusaurus v3 personal website for Ron Amosa (uncommonengineer.com). Digital garden with technical docs, blog essays, a newsletter, and a changelog system. Hosted on GitHub Pages with custom domain. Package spec: `@docusaurus/core: ^3.9.2` (in `website/package.json`).

The repo root holds only a stray `package.json`/`package-lock.json` and an automation `scripts/` directory â€” the actual site lives entirely under `website/`. Run all npm commands from `website/`.

## Development Commands

All commands run from the `website/` directory:

```bash
npm run start     # Dev server with hot reload
npm run build     # Production build (use to verify before committing)
npm run serve     # Serve production build locally
npm run deploy    # Deploy to GitHub Pages
npm run clear     # Clear Docusaurus cache (useful when builds behave unexpectedly)
```

No test suite or linter is configured.

## Architecture

### Three Docs Instances

The site runs three instances of `@docusaurus/plugin-content-docs`, each with its own content dir and sidebar file:

1. **Main docs** (preset-classic) â€” `website/docs/` â†’ `/docs/`, sidebar `sidebars.js`
2. **Changelog docs** (standalone plugin, id: `changelog`) â€” `website/changelog/` â†’ `/changelog/`, sidebar `sidebars-changelog.js`
3. **Hackathon docs** (standalone plugin, id: `hackathon`) â€” `website/hackathon/` â†’ `/hackathon/`, sidebar `sidebars-hackathon.js`

All use autogenerated sidebars from filesystem structure. All three set `showLastUpdate*`; only hackathon enables breadcrumbs.

### Redirects

`@docusaurus/plugin-client-redirects` handles URL migrations, configured in `docusaurus.config.js`:
- `redirects.js` â€” legacy static redirect map (imported as `legacyRedirects`)
- `ga-redirects.js` â€” large redirect table (present in repo; referenced for GA-tracked legacy paths)
- An inline `createRedirects(existingPath)` function auto-generates aliases (e.g. old `/docs/engineer/guides/*` â†’ `/docs/engineer/Misc/*`)

`trailingSlash: true` is enforced site-wide so Docusaurus emits canonical/og:url/sitemap URLs matching what GitHub Pages actually serves (it 301s non-slash URLs). Don't change this without understanding the canonical-URL implications documented in the config comments.

### Diagrams

`@docusaurus/theme-mermaid` is enabled (`markdown.mermaid: true`) â€” Mermaid code fences render as diagrams in any doc/blog content.

### Custom Plugins

`website/plugins/` holds two local plugins registered in `docusaurus.config.js`:

- `content-counts.js` â€” Counts markdown files in `blog/` and `docs/`, extracts the latest blog post title/slug, exposes them via `setGlobalData` for React pages (e.g. the homepage). Also generates `llms.txt` (the LLM content map linked from `headTags`).
- `docs-rss-feed.js` â€” Generates an RSS feed for docs content, using git commit timestamps (`git log -1 --format=%ct`) as item dates.

### Client Modules

Registered in `docusaurus.config.js` under `clientModules`, these fire GA4 custom events on every route change:

- `src/clientModules/blogEngagementTracker.js` â€” Tracks blog session depth (`blog_section_enter`, `blog_page_view`, `blog_deep_engagement` at 3+ pages)
- `src/clientModules/contentSectionTracker.js` â€” Tags each page with its content section (analysis, docs, newsletter, homepage, changelog, projects, about) for traffic source segmentation

## Content Conventions

### Blog Posts

Two filename patterns are in use:
- Older/technical posts: `YYYY-MM-DD-kebab-case-title.md` (e.g. `2025-03-04-agentic-ai-part-4.md`)
- Newer essay-style posts: `YYYY-MM-DD - Title With Spaces.md` (e.g. `2026-02-01 - Inevitability of AI.md`)

Blog posts are **experiential documentation** â€” they document the discovery process, not just solutions. Include:
- The journey and what was tried (including failed approaches)
- Real-world gotchas, edge cases, and unexpected challenges
- Decision rationale for why approaches were chosen
- Testing results with evidence

When content was co-written with AI assistance, include this admonition:

```markdown
:::info đź¤– AI Collaboration
This post was co-written with AI assistance. All technical testing, troubleshooting, and real-world insights are from the author's direct experience. AI helped with structure, clarity, and documentation formatting.
:::
```

### Documentation

Filename pattern: `feature-name-guide.md` (kebab-case)

Organized under `website/docs/` in category directories: `engineer/`, `hacker/`, `study/`, `books/`, `archive/`. There is also a `start-here.mdx` at the docs root as a landing/intro page.

### Frontmatter

All content files require proper frontmatter. Use relative links starting with `/docs/` or `/blog/`.

Blog posts should include the full set of frontmatter fields. Here is the standard template:

```yaml
slug: kebab-case-slug
title: "Post Title"
description: "Compelling 150-160 char description for search snippets â€” avoid generic filler."
date: YYYY-MM-DD
authors: [ron]
keywords:
  - 4-8 specific keywords
  - relevant to actual content
tags:
  - AI-Sovereignty-Governance
  - tech-politics
image: /img/blog/post-image.png
reading_time: 6
word_count: 1200
hide_table_of_contents: false
draft: false
```

The `authors.yml` file in `website/blog/` defines author metadata including social links (LinkedIn, GitHub, YouTube, Bluesky).

### Git Commit Messages

Follow conventional commits style:
- `docs: add {feature} documentation and update changelog`
- `feat: implement {feature} and document in changelog`
- `fix: resolve {issue} and update changelog`
