## Project Overview

Docusaurus v3 personal website for Ron Amosa (uncommonengineer.com). Digital garden with technical docs, blog essays, a newsletter, and a changelog system. Hosted on GitHub Pages with custom domain. Package spec: `@docusaurus/core: ^3.8.1`.

## Development Commands

All commands run from the `website/` directory:

```bash
npm run start     # Dev server with hot reload
npm run build     # Production build (use to verify before committing)
npm run serve     # Serve production build locally
npm run deploy    # Deploy to GitHub Pages
npm run clear     # Clear Docusaurus cache (useful when builds behave unexpectedly)
```

No test suite or linter is configured.

## Architecture

### Dual Docs Plugin

The site runs two instances of `@docusaurus/plugin-content-docs`:

1. **Main docs** (preset-classic) — `website/docs/` routed to `/docs/`
2. **Changelog docs** (standalone plugin, id: `changelog`) — `website/changelog/` routed to `/changelog/`

Both use autogenerated sidebars from filesystem structure. The changelog plugin has its own sidebar config in `sidebars-changelog.js`.

### Custom Plugin

`website/plugins/content-counts.js` — Counts markdown files in `blog/` and `docs/`, extracts the latest blog post title/slug, and exposes everything via `setGlobalData` for use in React pages (e.g. the homepage).

### Client Modules

Registered in `docusaurus.config.js` under `clientModules`, these fire GA4 custom events on every route change:

- `src/clientModules/blogEngagementTracker.js` — Tracks blog session depth (`blog_section_enter`, `blog_page_view`, `blog_deep_engagement` at 3+ pages)
- `src/clientModules/contentSectionTracker.js` — Tags each page with its content section (analysis, docs, newsletter, homepage, changelog, projects, about) for traffic source segmentation

### Custom Pages (React)

`website/src/pages/` contains custom React pages beyond the standard Docusaurus markdown:
- `index.js` — Homepage (uses `content-counts` global data)
- `about/` — About page
- `projects/` — Projects page (data in `src/data/_Projects.js`)
- `subscribe/` — Newsletter subscription (Beehiiv embed)
- `components/_SocialLinks.js` — Shared social links component

Data files in `src/data/` (`_Projects.js`, `_SocialLinks.js`) define structured content for these pages.

### Theme Customization

- `src/theme/BlogPostItem/` — Swizzled blog post component with custom styling
- `src/theme/DocItem/Footer/` — Swizzled doc footer with newsletter CTA ("Subscribe here" link to `/subscribe`) and GA4 `newsletter_cta_click` tracking
- `src/css/custom.css` — Global style overrides
- Dark mode only (`disableSwitch: true`, `respectPrefersColorScheme: false`)
- Prism theme: `nightOwl`, additional languages: Ruby, HCL, Docker, YAML
- Font Awesome icons via `@fortawesome` packages (fontawesome-svg-core, free-brands, free-regular, free-solid, react-fontawesome)

### Static Assets

- `static/img/` — All images (referenced as `/img/...` in content), includes `social-card.png` for Open Graph/Twitter
- `static/js/console-clue.js` — Easter egg for PTEC community
- `static/js/metricool.js` — Metricool analytics tracking
- `static/CNAME` — Custom domain (`www.uncommonengineer.com`)

### Build Configuration

- `onBrokenLinks: 'warn'` and `onBrokenMarkdownLinks: 'warn'` — builds won't fail on broken links, but warnings should be addressed
- Google Analytics via `@docusaurus/plugin-google-gtag` (tracking ID: `G-DMRNTVGLRC`, `anonymizeIP: true`)
- Algolia search (appId: `9UFF3RBJQ9`, index: `uncommonengineer.com`)
- Beehiiv newsletter embed script loaded globally (`subscribe-forms.beehiiv.com/embed.js`)
- Twitter card meta: `summary_large_image`
- Docs and blog both have `editUrl` pointing to the GitHub repo (`/edit/main/website/`)

## Changelog System

**Always update the changelog when making significant changes** (new docs, blog posts, config changes, new features). Skip for typo fixes or formatting-only changes.

Changelog files live in `website/changelog/{YEAR}/{month}.md`. Entries go at the TOP of the file (most recent first).

### Entry Format

```markdown
## 📅 YYYY-MM-DD

### {Category}
- **Title**: Brief description
  - Additional details
  - **Link**: [Link Text](/path/to/content)
```

Categories: 🛠️ Site Improvements, 📚 Documentation Updates, 📝 Blog Posts, 🔧 Bug Fixes, 🎨 Design Changes, ⚡ Performance Improvements, 🔒 Security Updates, 📦 Dependencies

### New Month File Template

If the current month's file doesn't exist, create it at `website/changelog/{YEAR}/{month}.md`:

```markdown
---
sidebar_position: 1
title: {Month YYYY}
description: Changes and additions for {Month YYYY}
---

# {Month YYYY}

## 📅 YYYY-MM-DD

### {CATEGORY}
- **Your Change**: Description
```

## Content Conventions

### Blog Posts

Two filename patterns are in use:
- Older/technical posts: `YYYY-MM-DD-kebab-case-title.md` (e.g. `2025-03-04-agentic-ai-part-4.md`)
- Newer essay-style posts: `YYYY-MM-DD - Title With Spaces.md` (e.g. `2026-02-01 - Inevitability of AI.md`)

Blog posts are **experiential documentation** — they document the discovery process, not just solutions. Include:
- The journey and what was tried (including failed approaches)
- Real-world gotchas, edge cases, and unexpected challenges
- Decision rationale for why approaches were chosen
- Testing results with evidence

When content was co-written with AI assistance, include this admonition:

```markdown
:::info 🤖 AI Collaboration
This post was co-written with AI assistance. All technical testing, troubleshooting, and real-world insights are from the author's direct experience. AI helped with structure, clarity, and documentation formatting.
:::
```

### Documentation

Filename pattern: `feature-name-guide.md` (kebab-case)

Organized under `website/docs/` in category directories: `engineer/`, `hacker/`, `study/`, `books/`, `archive/`. There is also a `start-here.md` at the docs root as a landing/intro page.

### Frontmatter

All content files require proper frontmatter. Use relative links starting with `/docs/` or `/blog/`.

### Git Commit Messages

Follow conventional commits style:
- `docs: add {feature} documentation and update changelog`
- `feat: implement {feature} and document in changelog`
- `fix: resolve {issue} and update changelog`
