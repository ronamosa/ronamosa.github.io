# Design Document: Hackathon Docs

## Overview

This design adds a self-contained `/hackathon` documentation section to the existing Docusaurus v3 site at uncommonengineer.com. The section is implemented as a separate `@docusaurus/plugin-content-docs` instance — the same pattern already used by the `/changelog` section — giving it independent routing, sidebar, and content management without touching any existing site content.

The hackathon section contains eight MDX pages serving four audiences (students, mentors/facilitators, school IT administrators, and event judges) for the "Tech for Good Hackathon 2026" across Wellington, Hamilton, and Auckland.

### Design Decisions

1. **Separate plugin instance over nested docs folder**: A dedicated `@docusaurus/plugin-content-docs` instance (id: `hackathon`) keeps the hackathon content completely isolated from the main `/docs` section. This mirrors the existing `changelog` plugin pattern, avoids polluting the main docs sidebar, and makes the section easy to remove after the event.

2. **Autogenerated sidebar**: Using `{type: 'autogenerated', dirName: '.'}` with `sidebar_position` frontmatter in each MDX file. This is the same approach used by both the main docs and changelog sidebars, keeping the pattern consistent and requiring no manual sidebar maintenance.

3. **MDX over plain Markdown**: All pages use `.mdx` extension to enable Docusaurus components (Tabs, Admonitions) inline. The site already uses MDX (e.g., `start-here.mdx`), so this is consistent.

4. **No custom CSS or external dependencies**: The hackathon section relies entirely on existing Docusaurus theme components and styling. No new npm packages, no custom CSS files.

5. **Frontmatter-driven ordering**: Each page uses `sidebar_position` in its frontmatter to control sidebar order, which is the standard Docusaurus approach for autogenerated sidebars.

6. **Trustworthy AI delivered inline on the Student Guide, not as a standalone page**: The responsible-AI content lives as a subsection of `get-started.mdx` (with a stable anchor for fragment links) plus a short mentor-coaching note in `mentor-guide.mdx`. Rationale: the people who most need this guidance — students about to prompt Kiro, and the mentors coaching them — are already on those pages, so reading-in-place lands the content at the moment of use. The section's navigation is also already dense with eight pages; a ninth navigation entry for a topic that's more guidance than reference would add noise for little return. This decision aligns with Requirement 11 AC 6, which explicitly prohibits a separate top-level Trustworthy AI page.

7. **Judging Guide as a dedicated page, not appended to the Mentor Guide**: Judges are a distinct fourth audience with their own information needs (assessment criteria, feedback framing, body language, supporting student confidence). A standalone `judging-guide.mdx` keeps that guidance clean, shareable with non-mentor judges, and linkable from both the Landing Page and the Mentor Guide without forcing judges to read mentor-only content. The Mentor Guide retains a short "What Judges Look For" summary that links through. This decision is driven by Requirement 12.

### Navbar Integration

The "Hackathon" link is added to the existing `navbar.items` array in `docusaurus.config.js`, positioned on the left alongside "Technical" and "Analysis & Essays":

```
[Technical] [Analysis & Essays] [Hackathon]          [About] [Projects] [Changelog]
```

The navbar item uses `to: "hackathon/"` with `activeBasePath: "hackathon"` so it highlights when browsing any page within the hackathon section. Adding the fourth audience (judges) does not change the navbar — judges reach the Judging Guide via the Landing Page CTA or a direct link. The single "Hackathon" navbar entry remains the only top-level hook into the section.

## Components and Interfaces

### Configuration Files

#### 1. `docusaurus.config.js` — Plugin Addition

A new `@docusaurus/plugin-content-docs` entry is added to the `plugins` array:

```javascript
{
  id: 'hackathon',
  path: 'hackathon',
  routeBasePath: 'hackathon',
  sidebarPath: require.resolve('./sidebars-hackathon.js'),
  editUrl: 'https://github.com/ronamosa/ronamosa.github.io/edit/main/website/',
  showLastUpdateAuthor: true,
  showLastUpdateTime: true,
  breadcrumbs: true,
}
```

This mirrors the existing changelog plugin configuration. Key differences from changelog: `breadcrumbs: true` (changelog has `breadcrumbs: false`) since hackathon pages benefit from breadcrumb navigation.

#### 2. `docusaurus.config.js` — Navbar Item

A new item is inserted into `navbar.items` at position index 2 (after "Analysis & Essays", before "About"):

```javascript
{
  to: 'hackathon/',
  activeBasePath: 'hackathon',
  label: 'Hackathon',
  position: 'left',
}
```

#### 3. `sidebars-hackathon.js` — Sidebar Configuration

A new file following the exact pattern of `sidebars-changelog.js`:

```javascript
module.exports = {
  hackathonSidebar: [{type: 'autogenerated', dirName: '.'}],
};
```

### Content Files

All content lives in `website/hackathon/`. Each file is an MDX file with frontmatter controlling its sidebar position and metadata.

| File | Route | `sidebar_position` | Audience |
|---|---|---|---|
| `index.mdx` | `/hackathon/` | 1 | All |
| `setup.mdx` | `/hackathon/setup` | 2 | IT Admins |
| `get-started.mdx` | `/hackathon/get-started` | 3 | Students |
| `prompts.mdx` | `/hackathon/prompts` | 4 | Students |
| `troubleshooting.mdx` | `/hackathon/troubleshooting` | 5 | All |
| `mentor-guide.mdx` | `/hackathon/mentor-guide` | 6 | Mentors |
| `judging-guide.mdx` | `/hackathon/judging-guide` | 7 | Judges |
| `resources.mdx` | `/hackathon/resources` | 8 | All |

Ordering rationale: pages are grouped by audience flow. Students come first (landing → IT setup → student guide → prompts → troubleshooting), then the two on-the-day facilitator roles sit adjacent (mentor guide → judging guide), with the all-audience resources page last. Placing the Judging Guide between the Mentor Guide and Resources keeps the two facilitator roles together in the sidebar and pushes the reference-style Resources page to the tail, where readers expect a "links appendix" to live.

### MDX Components Used

All components are built into Docusaurus — no imports needed for Admonitions, and only standard Docusaurus imports for Tabs:

| Component | Usage | Import Required |
|---|---|---|
| Admonitions (`:::tip`, `:::warning`, `:::note`, `:::info`, `:::danger`) | Tips, warnings, notes across all pages | No (native MDX syntax) |
| `Tabs` / `TabItem` | Platform-specific instructions (macOS vs Windows) on setup and troubleshooting pages | Yes: `import Tabs from '@theme/Tabs'; import TabItem from '@theme/TabItem';` |
| Fenced code blocks | Prompt examples, CLI commands (with copy button built-in) | No |

- **Contacts section (new, aligned with Requirements 3 AC 9 and 13)**: Lists both Regional Points of Contact with name, region coverage, and email. Placeholders use the format `[name — TBC]` / `[email — TBC]` until final values are supplied:
  - Auckland and Hamilton — `[name — TBC]`, `[email — TBC]`
  - Wellington — Shane `[surname — TBC]`, `[email — TBC]`
- **Trustworthy AI lead-in (new, aligned with Requirements 3 AC 8 and 11 AC 4)**: A short paragraph introducing the event's stance on responsible AI use, with a fragment link into the Student Guide's Trustworthy AI subsection (e.g., `/hackathon/get-started#trustworthy-ai`). No duplicated content on this page — the authoritative text lives on the Student Guide.
- **Asset note**: The existing Hynds Foundation logo file at `/img/hackathon/hynds-foundation-logo.png` has been replaced in-place with the final asset, so no MDX path change is required.
- **Components**: Admonitions for key dates and for the Contacts block.

#### `setup.mdx` — Device Setup Guide

- **Content**: Numbered pre-event checklist, system requirements, "Verify it works" section, and a named Regional Points of Contact block.
- **Regional Points of Contact (replaces placeholder support contact, aligned with Requirements 4 AC 6 and 13)**: The existing "Organiser Contact — Placeholder" admonition near the bottom of the page is the slot that will be replaced. The generic "placeholder for support contact" wording is superseded by a Contacts block listing:
  - Auckland and Hamilton — `[name — TBC]`, `[email — TBC]`
  - Wellington — Shane `[surname — TBC]`, `[email — TBC]`
  Placeholders render with a visible `[… — TBC]` marker so the gap is obvious during review and is not missed before go-live.
- **Components**: `:::warning` for ChromeOS incompatibility, `:::info` for "Kiro is free" note, `Tabs` for macOS/Windows install steps, and a `:::note` admonition hosting the Regional Points of Contact block.

#### `get-started.mdx` — Student Guide

- **Content**: Numbered stage-by-stage walkthrough of the day (welcome → problem selection → empathise → ideate → build with Kiro → test & iterate → pitch prep → closing), plus a Trustworthy AI subsection.
- **Order of Events table (aligned with Requirement 5 AC 5)**: The current 10-row schedule table is replaced with the stakeholder-supplied **13-row Order of Events**:
  1. 9:00–9:30 am — School registration (Deloitte Reception)
  2. 9:30–9:40 am — Session welcome and introduction (10 min)
  3. 9:40–10:25 am — Icebreaker and problem exploration (45 min)
  4. 10:25–11:00 am — Ideation (30 min)
  5. 11:00–11:10 am — Short break (10 min)
  6. 11:10 am–12:10 pm — Design and prototyping ideas (60 min)
  7. 12:10–12:35 pm — Lunch (25 min)
  8. 12:35–1:00 pm — Final testing and refine pitch presentations (25 min)
  9. 1:00–1:50 pm — Pitch presentations and judges feedback (50 min, 6 min per team)
  10. 1:50–2:05 pm — Judges deliberation and student energizer (15 min)
  11. 2:05–2:15 pm — Winners announced and prizes (10 min)
  12. 2:15–2:30 pm — Session reflection and wrap-up (15 min)
  13. 2:30–2:45 pm — Schools depart (15 min)
- **Icebreaker rename (aligned with Requirements 5 AC 6 and 5 AC 7)**: The warm-up activity is labelled "Icebreaker" throughout the page and described in terms consistent with the 9:40–10:25 am "Icebreaker and problem exploration" block. **No scavenger hunt reference anywhere** — that wording is removed.
- **Trustworthy AI subsection (new, aligned with Requirement 11)**: A substantive subsection with a stable anchor (e.g., `## Trustworthy AI {#trustworthy-ai}`) so the Landing Page and Mentor Guide can deep-link to it. Covers the four topics from Requirement 11 AC 2 as clearly labelled callouts or subheadings:
  - Honesty about AI use (openly acknowledge AI assistance in the pitch)
  - Verification of AI-generated output (cross-check facts, code, and claims)
  - Privacy and safety with prompts (do not paste personal information into the Kiro AI Chatbot)
  - What to do if something feels wrong (stop and ask a mentor if output feels harmful, unsafe, or inappropriate)
- **Trustworthy AI pointer (aligned with Requirement 5 AC 9)**: A `:::tip` or `:::warning` admonition placed before the Build with Kiro stage directs students to read the Trustworthy AI subsection before they start prompting.
- **Design and Prototyping stage (aligned with Requirement 5 AC 8)**: Explicitly identifies the Kiro AI Chatbot as the tool used to design, build, and iterate on the solution during the 11:10 am–12:10 pm block.
- **Components**: `:::tip` admonitions for encouragement, `:::warning` for common pitfalls, link to the Troubleshooting page, and a link to the Trustworthy AI subsection anchor.

#### `prompts.mdx` — Prompt Library

- **Content**: Five themed sections (Ocean/Water Pollution, Youth Wellbeing, Community Food Sharing, Native Species Conservation, School Sustainability), each with starter/iteration/stretch prompts in code blocks. "What makes a good prompt" section with good vs vague examples.
- **Components**: Fenced code blocks for all prompts.

#### `troubleshooting.mdx` — Troubleshooting

- **Content**: Problem → Solution format for 8 common issues.
- **Components**: `Tabs` for platform-specific fixes (macOS Gatekeeper vs Windows SmartScreen), `:::tip` for quick fixes.

#### `mentor-guide.mdx` — Mentor Playbook

- **Content**: Role description, day-at-a-glance stage guidance, first-5-minutes triage decision tree, prompt coaching tips, conversation starters, a summarised "What Judges Look For" section, and a short mentor-coaching note on Trustworthy AI.
- **Day at a Glance table (aligned with Requirement 8 AC 2)**: Stage names and sequence must match the Student Guide's 13-row Order of Events **exactly**. Times and durations may be collapsed into a single "when" column for brevity, but the stage names themselves are canonical and must match. This keeps mentors and students anchored to the same mental model of the day.
- **No scavenger hunt (aligned with Requirement 8 AC 3)**: All scavenger hunt references are removed. The warm-up stage is labelled "Icebreaker" consistent with the Student Guide.
- **What Judges Look For section (aligned with Requirements 8 AC 7 and 12 AC 11)**: Becomes a short summary (a few sentences or a tight bulleted list naming the six criteria) with a prominent link to `/hackathon/judging-guide` as the authoritative source. Full criteria descriptions, feedback framing, and body-language guidance are **not** duplicated here.
- **Trustworthy AI coaching note (new, aligned with Requirements 8 AC 8 and 11 AC 3)**: A short admonition or subsection explaining how mentors reinforce the four Trustworthy AI topics with their teams during the Design and Prototyping stage. Links to the Student Guide's Trustworthy AI subsection anchor (`/hackathon/get-started#trustworthy-ai`) as the authoritative content rather than duplicating it.
- **Components**: `:::tip` for coaching advice, `:::note` for timing reminders, Mermaid diagram for the triage decision tree.

#### `judging-guide.mdx` — Judging Guide (new page)

- **Frontmatter**:
  - `title: "Judging Guide"`
  - `description`: short summary of the page's purpose for judges, mentors, and curious students
  - `sidebar_position: 7`
  - `sidebar_label: "Judging Guide"`
- **Access**: Publicly accessible, no authentication. No new dependencies.
- **Sections (aligned with Requirement 12 AC 3–9)**:
  1. **Judging Purpose** — Opening section stating that judging focuses on student learning, effort, and potential impact; judges assess the idea, the thinking, and the clarity rather than visual polish; and judges reward purpose, creativity, and understanding.
  2. **What to Look For** — Describes the six assessment criteria. Each criterion gets a short intro plus the descriptive sub-bullets sourced from the stakeholder-supplied judging guidance document:
     - Problem and Impact
     - Solution
     - Use of Kiro
     - Feasibility
     - Creativity and Innovation
     - Communication
  3. **Giving Positive, Constructive Feedback** — Covers:
     - Strengths-first framing
     - "Next step" language rather than criticism
     - A two-positives-to-one-suggestion ratio
     - Feedback on the idea, not the person
     - Curious questions rather than "gotcha" questions
     - At least two example phrasings that judges can reuse verbatim
  4. **Facial Expressions and Body Language** — Smiling, nodding, engaged posture; save critique for Q&A rather than showing it during the pitch itself.
  5. **Supporting Student Confidence** — Calm, warm, encouraging tone; thanking teams for presenting; simple affirmations.
  6. **Final Reminder** — Students will remember how it felt to present more than what was said.
- **Components**: Admonitions consistent with the rest of the section (`:::tip`, `:::note`, `:::info`). No Tabs or Mermaid required.
- **Inbound links**: Linked from the Landing Page "I'm a judge" CTA (Requirement 3 AC 6) and from the Mentor Guide "What Judges Look For" section (Requirement 8 AC 7).

#### `resources.mdx` — Resources Page

- **Content**: Themed starter repo links (5), ZIP download links, external resource links (pitch template, judging criteria, Kiro docs, Node.js downloads, event schedule).
- **Components**: `:::info` for placeholder URL notices.

## Data Models

This feature introduces no runtime data models, databases, or APIs. All content is static MDX rendered at build time by Docusaurus.

### Frontmatter Schema

Each MDX file uses the following frontmatter fields:

```yaml
---
title: string          # Page title displayed in sidebar and browser tab
description: string    # Meta description for SEO
sidebar_position: int  # Order in the autogenerated sidebar (1-8)
sidebar_label: string  # Optional shorter label for sidebar display
slug: string           # Optional, only used on index.mdx (slug: /)
---
```

The `sidebar_position` range is now **1–8** (previously 1–7) to accommodate the new `judging-guide.mdx`. No other schema changes.

## Error Handling

Since this feature is entirely static documentation with no runtime logic, error handling focuses on build-time and content integrity:

| Scenario | Handling |
|---|---|
| Broken internal links between hackathon pages | Docusaurus `onBrokenLinks: 'warn'` (existing config) will flag these during build |
| Broken markdown links | Docusaurus `onBrokenMarkdownLinks: 'warn'` (existing config) will flag these during build |
| Broken fragment/anchor links (e.g., `#trustworthy-ai`) | Surface as build-time warnings; must be verified during manual smoke testing since anchor resolution is page-scoped |
| Missing MDX imports (Tabs/TabItem) | Docusaurus build will fail with a clear error pointing to the file and missing import |
| Invalid frontmatter | Docusaurus build will fail with a parse error identifying the file |
| Missing `hackathon/` directory | Docusaurus build will fail because the plugin's `path` doesn't resolve — caught immediately on first build |
| Placeholder URLs (not yet replaced) | Use `#` as href with a visible `:::info` admonition noting the link is a placeholder — no broken link warnings |
| Placeholder names/emails for Regional Points of Contact | Rendered as visible `[name — TBC]` / `[email — TBC]` markers so missing values are obvious during review and before go-live |

## Testing Strategy

### Why Property-Based Testing Does Not Apply

This feature consists entirely of:

- Declarative Docusaurus configuration (plugin instance, navbar item, sidebar)
- Static MDX content files with no custom logic

There are no pure functions, data transformations, parsers, or business logic to test with property-based testing. The "inputs" are fixed configuration and static content, not a variable input space.

### Testing Approach

**1. Build Verification (Primary)**

- Run `docusaurus build` to verify the entire site compiles without errors
- This validates: plugin configuration, sidebar resolution, MDX parsing, internal link integrity, component imports
- Command: `npm run build` from the `website/` directory

**2. Manual Smoke Testing**

- Run `docusaurus start` and verify:
  - `/hackathon/` loads the landing page
  - All 8 pages are accessible via their routes
  - `/hackathon/judging-guide` loads and renders all six assessment criteria correctly, including the descriptive sub-bullets under each
  - Sidebar displays all pages in correct order (1 → 8), with the Judging Guide sitting between the Mentor Guide and Resources
  - Navbar "Hackathon" link appears and highlights correctly on every hackathon page
  - Landing page displays **four** audience CTAs (student, mentor/facilitator, IT, judge) and the judge CTA resolves to `/hackathon/judging-guide`
  - Landing page and Setup Guide both display a Regional Points of Contact block with the two named contacts and visible `[… — TBC]` placeholders where values are pending
  - Fragment link from the Landing Page to the Student Guide's Trustworthy AI anchor (`/hackathon/get-started#trustworthy-ai`) resolves and scrolls to the correct section with no broken-anchor warning
  - Fragment link from the Mentor Guide's Trustworthy AI coaching note to the same anchor resolves correctly
  - Tabs component switches between macOS/Windows content
  - Admonitions render with correct styling
  - Code blocks have copy buttons
  - Pages render correctly on mobile viewport

**3. Content Review Checklist**

- Each page addresses its target audience in the appropriate tone
- All placeholder URLs are marked with `:::info` admonitions
- Partner names and event details match requirements — specifically the Landing Page partner wording matches Requirement 3 AC 3 (Hynds Foundation tagged as "Funding and Programme Delivery Partner")
- Prize funding wording on the Landing Page matches Requirement 3 AC 4 (jointly funded by Deloitte, AWS, and the Hynds Foundation)
- All 5 problem themes are present in the Prompt Library
- Troubleshooting covers all 8 required issues
- Judging Guide covers all six assessment criteria (Problem and Impact, Solution, Use of Kiro, Feasibility, Creativity and Innovation, Communication) with the descriptive sub-bullets from the stakeholder-supplied judging guidance document
- Judging Guide includes the Giving Positive, Constructive Feedback section with the 2:1 ratio, "next step" language, and at least two reusable example phrasings
- Judging Guide includes Facial Expressions and Body Language, Supporting Student Confidence, and the Final Reminder sections
- Student Guide's Trustworthy AI subsection covers all four Requirement 11 AC 2 topics (honesty, verification, privacy/safety, what to do if something feels wrong)
- Mentor Guide's Trustworthy AI coaching note links to the Student Guide's anchor and does not duplicate the substantive content
- Mentor Guide's "What Judges Look For" section is a summary that links to the Judging Guide rather than duplicating it
- **No "scavenger hunt" reference anywhere** in the hackathon section — Student Guide, Mentor Guide, or otherwise
- Student Guide's day schedule matches the Order of Events exactly (13 rows, stage names as specified in Requirement 5 AC 5)
- Mentor Guide's "Day at a Glance" table stage names and sequence match the Student Guide's Order of Events exactly
- Warm-up activity is labelled "Icebreaker" consistently across the Student Guide and Mentor Guide and aligns with the 9:40–10:25 am "Icebreaker and problem exploration" block

**4. Non-Regression Verification**

- Confirm existing `/docs`, `/blog`, and `/changelog` routes still work
- Confirm `sidebars.js` and `sidebars-changelog.js` are unmodified
- Confirm no new npm dependencies were added
