# Design Document: Hackathon Docs

## Overview

This design adds a self-contained `/hackathon` documentation section to the existing Docusaurus v3 site at uncommonengineer.com. The section is implemented as a separate `@docusaurus/plugin-content-docs` instance — the same pattern already used by the `/changelog` section — giving it independent routing, sidebar, and content management without touching any existing site content.

The hackathon section contains seven MDX pages serving three audiences (students, mentors, IT administrators) for the "Tech for Good Hackathon 2026" across Wellington, Hamilton, and Auckland.

### Design Decisions

1. **Separate plugin instance over nested docs folder**: A dedicated `@docusaurus/plugin-content-docs` instance (id: `hackathon`) keeps the hackathon content completely isolated from the main `/docs` section. This mirrors the existing `changelog` plugin pattern, avoids polluting the main docs sidebar, and makes the section easy to remove after the event.

2. **Autogenerated sidebar**: Using `{type: 'autogenerated', dirName: '.'}` with `sidebar_position` frontmatter in each MDX file. This is the same approach used by both the main docs and changelog sidebars, keeping the pattern consistent and requiring no manual sidebar maintenance.

3. **MDX over plain Markdown**: All pages use `.mdx` extension to enable Docusaurus components (Tabs, Admonitions) inline. The site already uses MDX (e.g., `start-here.mdx`), so this is consistent.

4. **No custom CSS or external dependencies**: The hackathon section relies entirely on existing Docusaurus theme components and styling. No new npm packages, no custom CSS files.

5. **Frontmatter-driven ordering**: Each page uses `sidebar_position` in its frontmatter to control sidebar order, which is the standard Docusaurus approach for autogenerated sidebars.

### Navbar Integration

The "Hackathon" link is added to the existing `navbar.items` array in `docusaurus.config.js`, positioned on the left alongside "Technical" and "Analysis & Essays":

```
[Technical] [Analysis & Essays] [Hackathon]          [About] [Projects] [Changelog]
```

The navbar item uses `to: "hackathon/"` with `activeBasePath: "hackathon"` so it highlights when browsing any page within the hackathon section.

## Components and Interfaces

### Configuration Files

#### 1. `docusaurus.config.js` — Plugin Addition

A new `@docusaurus/plugin-content-docs` entry is added to the `plugins` array:

```javascript
{
  id: 'hackathon',
  path: 'hackathon',
  routeBasePath: 'hackathon',
  sidebarPath: require.resolve('./sidebars-hackathon.js'),
  editUrl: 'https://github.com/ronamosa/ronamosa.github.io/edit/main/website/',
  showLastUpdateAuthor: true,
  showLastUpdateTime: true,
  breadcrumbs: true,
}
```

This mirrors the existing changelog plugin configuration. Key differences from changelog: `breadcrumbs: true` (changelog has `breadcrumbs: false`) since hackathon pages benefit from breadcrumb navigation.

#### 2. `docusaurus.config.js` — Navbar Item

A new item is inserted into `navbar.items` at position index 2 (after "Analysis & Essays", before "About"):

```javascript
{
  to: 'hackathon/',
  activeBasePath: 'hackathon',
  label: 'Hackathon',
  position: 'left',
}
```

#### 3. `sidebars-hackathon.js` — Sidebar Configuration

A new file following the exact pattern of `sidebars-changelog.js`:

```javascript
module.exports = {
  hackathonSidebar: [{type: 'autogenerated', dirName: '.'}],
};
```

### Content Files

All content lives in `website/hackathon/`. Each file is an MDX file with frontmatter controlling its sidebar position and metadata.

| File | Route | `sidebar_position` | Audience |
|---|---|---|---|
| `index.mdx` | `/hackathon/` | 1 | All |
| `setup.mdx` | `/hackathon/setup` | 2 | IT Admins |
| `get-started.mdx` | `/hackathon/get-started` | 3 | Students |
| `prompts.mdx` | `/hackathon/prompts` | 4 | Students |

| `resources.mdx` | `/hackathon/resources` | 7 | All |

### MDX Components Used

All components are built into Docusaurus — no imports needed for Admonitions, and only standard Docusaurus imports for Tabs:

| Component | Usage | Import Required |
|---|---|---|
| Admonitions (`:::tip`, `:::warning`, `:::note`, `:::info`, `:::danger`) | Tips, warnings, notes across all pages | No (native MDX syntax) |
| `Tabs` / `TabItem` | Platform-specific instructions (macOS vs Windows) on setup and troubleshooting pages | Yes: `import Tabs from '@theme/Tabs'; import TabItem from '@theme/TabItem';` |
| Fenced code blocks | Prompt examples, CLI commands (with copy button built-in) | No |

### Page Content Specifications

#### `index.mdx` — Landing Page
- **Frontmatter**: `slug: /` to serve as the section root
- **Content**: Event title, partner logos (placeholder images), event dates/cities table, three audience call-to-action cards linking to get-started, mentor-guide, and setup pages
- **Components**: Admonitions for key dates

#### `setup.mdx` — Device Setup Guide
- **Content**: Numbered pre-event checklist, system requirements, "Verify it works" section, support contact placeholder
- **Components**: `:::warning` for ChromeOS incompatibility, `:::info` for "Kiro is free" note, `Tabs` for macOS/Windows install steps

#### `get-started.mdx` — Student Guide
- **Content**: Numbered hackathon-day steps (open Kiro → open project → read requirements → prompt → watch code generate → preview → iterate → screenshot for pitch)
- **Components**: `:::tip` for encouragement, `:::warning` for common pitfalls, link to troubleshooting page

#### `prompts.mdx` — Prompt Library
- **Content**: Five themed sections (Ocean/Water Pollution, Youth Wellbeing, Community Food Sharing, Native Species Conservation, School Sustainability), each with starter/iteration/stretch prompts in code blocks. "What makes a good prompt" section with good vs vague examples.
- **Components**: Fenced code blocks for all prompts

#### `troubleshooting.mdx` — Troubleshooting
- **Content**: Problem → Solution format for 8 common issues
- **Components**: `Tabs` for platform-specific fixes (macOS Gatekeeper vs Windows SmartScreen), `:::tip` for quick fixes

#### `mentor-guide.mdx` — Mentor Playbook
- **Content**: Role description, 40-minute build-phase timeline, first-5-minutes triage decision tree, prompt coaching tips, conversation starters, judging criteria overview
- **Components**: `:::tip` for coaching advice, `:::note` for timing reminders, Mermaid diagram for triage decision tree

#### `resources.mdx` — Resources Page
- **Content**: Themed starter repo links (5), ZIP download links, external resource links (pitch template, judging criteria, Kiro docs, Node.js downloads, event schedule)
- **Components**: `:::info` for placeholder URL notices

## Data Models

This feature introduces no runtime data models, databases, or APIs. All content is static MDX rendered at build time by Docusaurus.

### Frontmatter Schema

Each MDX file uses the following frontmatter fields:

```yaml
---
title: string          # Page title displayed in sidebar and browser tab
description: string    # Meta description for SEO
sidebar_position: int  # Order in the autogenerated sidebar (1-7)
sidebar_label: string  # Optional shorter label for sidebar display
slug: string           # Optional, only used on index.mdx (slug: /)
---
```

## Error Handling

Since this feature is entirely static documentation with no runtime logic, error handling focuses on build-time and content integrity:

| Scenario | Handling |
|---|---|
| Broken internal links between hackathon pages | Docusaurus `onBrokenLinks: 'warn'` (existing config) will flag these during build |
| Broken markdown links | Docusaurus `onBrokenMarkdownLinks: 'warn'` (existing config) will flag these during build |
| Missing MDX imports (Tabs/TabItem) | Docusaurus build will fail with a clear error pointing to the file and missing import |
| Invalid frontmatter | Docusaurus build will fail with a parse error identifying the file |
| Missing `hackathon/` directory | Docusaurus build will fail because the plugin's `path` doesn't resolve — caught immediately on first build |
| Placeholder URLs (not yet replaced) | Use `#` as href with a visible `:::info` admonition noting the link is a placeholder — no broken link warnings |

## Testing Strategy

### Why Property-Based Testing Does Not Apply

This feature consists entirely of:
- Declarative Docusaurus configuration (plugin instance, navbar item, sidebar)
- Static MDX content files with no custom logic

There are no pure functions, data transformations, parsers, or business logic to test with property-based testing. The "inputs" are fixed configuration and static content, not a variable input space.

### Testing Approach

**1. Build Verification (Primary)**
- Run `docusaurus build` to verify the entire site compiles without errors
- This validates: plugin configuration, sidebar resolution, MDX parsing, internal link integrity, component imports
- Command: `npm run build` from the `website/` directory

**2. Manual Smoke Testing**
- Run `docusaurus start` and verify:
  - `/hackathon/` loads the landing page
  - All 7 pages are accessible via their routes
  - Sidebar displays all pages in correct order
  - Navbar "Hackathon" link appears and highlights correctly
  - Tabs component switches between macOS/Windows content
  - Admonitions render with correct styling
  - Code blocks have copy buttons
  - Pages render correctly on mobile viewport

**3. Content Review Checklist**
- Each page addresses its target audience in the appropriate tone
- All placeholder URLs are marked with `:::info` admonitions
- Partner names and event details match requirements
- All 5 problem themes are present in the prompt library
- Troubleshooting covers all 8 required issues

**4. Non-Regression Verification**
- Confirm existing `/docs`, `/blog`, and `/changelog` routes still work
- Confirm `sidebars.js` and `sidebars-changelog.js` are unmodified
- Confirm no new npm dependencies were added
