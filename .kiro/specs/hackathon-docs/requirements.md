# Requirements Document

## Introduction

This feature adds a self-contained `/hackathon` documentation section to the existing Docusaurus v3 site at uncommonengineer.com. The section supports the "Tech for Good Hackathon 2026" — a youth hackathon for high school students across three New Zealand cities (Wellington, Hamilton, Auckland), run in partnership with AWS, Deloitte, and the Hynds Foundation. Students work in teams to pick a social or environmental problem and build a tech solution using Kiro, AWS's spec-driven AI IDE.

The hackathon docs serve three distinct audiences: students (the primary participants), school IT administrators (who prepare devices), and adult mentors/facilitators. The section must be mobile-friendly, use existing Docusaurus features (admonitions, tabs, code blocks), and not modify any existing site content.

## Glossary

- **Hackathon_Section**: The new `/hackathon` documentation area added to the Docusaurus site as a separate docs plugin instance with its own sidebar and content directory.
- **Landing_Page**: The root page at `/hackathon/` that provides an overview of the event and audience-specific entry points.
- **Setup_Guide**: The page at `/hackathon/setup` targeting school IT administrators with device preparation instructions.
- **Student_Guide**: The page at `/hackathon/get-started` providing step-by-step instructions for students on hackathon day.
- **Prompt_Library**: The page at `/hackathon/prompts` containing copy-paste prompt examples organised by problem theme.
- **Troubleshooting_Page**: The page at `/hackathon/troubleshooting` listing common issues and their solutions.
- **Mentor_Guide**: The page at `/hackathon/mentor-guide` containing the facilitator playbook for adult mentors.
- **Resources_Page**: The page at `/hackathon/resources` providing links to repos, templates, and external tools.
- **Navbar**: The top navigation bar of the Docusaurus site.
- **Sidebar**: The left-hand navigation panel specific to the Hackathon_Section.
- **Docusaurus_Plugin_Instance**: A separate `@docusaurus/plugin-content-docs` configuration that creates an independent docs section with its own content path, route base path, and sidebar.
- **Admonition**: A Docusaurus callout component used for tips, warnings, notes, and other highlighted content blocks.
- **Tab_Component**: A Docusaurus component that allows content to be toggled between labelled tabs (e.g., macOS vs Windows).
- **Kiro**: AWS's spec-driven AI IDE used by students during the hackathon to generate specs and code.
- **Starter_Repo**: A pre-configured project repository (one per problem theme) that students open in Kiro as their starting point.

## Requirements

### Requirement 1: Hackathon Docs Plugin Instance

**User Story:** As a site maintainer, I want the hackathon content to live in a separate Docusaurus docs plugin instance, so that the hackathon section has its own sidebar and routing without affecting existing site content.

#### Acceptance Criteria

1. THE Hackathon_Section SHALL be configured as a new `@docusaurus/plugin-content-docs` instance in `docusaurus.config.js` with a unique plugin id, a content path pointing to a `hackathon` directory, and a route base path of `hackathon`.
2. THE Hackathon_Section SHALL have its own dedicated sidebar configuration file separate from the main docs sidebar and the changelog sidebar.
3. THE Hackathon_Section SHALL use autogenerated sidebar entries from the `hackathon` content directory.
4. WHEN the Docusaurus site is built, THE Hackathon_Section SHALL produce pages routed under the `/hackathon/` URL path.
5. THE Hackathon_Section SHALL NOT modify any existing docs, blog, or changelog content, configuration, or sidebar files.

### Requirement 2: Navbar Integration

**User Story:** As a site visitor, I want a "Hackathon" link in the site navigation bar, so that I can discover and access the hackathon section from any page.

#### Acceptance Criteria

1. THE Navbar SHALL include a "Hackathon" link that navigates to the `/hackathon/` route.
2. THE Navbar "Hackathon" link SHALL be positioned on the left side of the navigation bar alongside the existing "Technical" and "Analysis & Essays" links.
3. WHEN a visitor is browsing any page within the Hackathon_Section, THE Navbar "Hackathon" link SHALL appear as the active navigation item.

### Requirement 3: Landing Page

**User Story:** As a hackathon participant, mentor, or IT administrator, I want a landing page that clearly identifies the event and directs me to the content relevant to my role, so that I can quickly find what I need.

#### Acceptance Criteria

1. THE Landing_Page SHALL display the event title "Tech for Good Hackathon 2026".
2. THE Landing_Page SHALL list the partner organisations: AWS, Deloitte, and Hynds Foundation, with placeholder references for partner logos.
3. THE Landing_Page SHALL display the event dates and cities: Wellington (May 18), Hamilton (May 19), Auckland (May 22).
4. THE Landing_Page SHALL provide three audience-specific entry points with clear call-to-action labels:
   - "I'm a student" linking to `/hackathon/get-started`
   - "I'm a mentor/facilitator" linking to `/hackathon/mentor-guide`
   - "I'm school IT" linking to `/hackathon/setup`
5. THE Landing_Page SHALL be the index page of the Hackathon_Section, served at the `/hackathon/` route.

### Requirement 4: Device Setup Guide

**User Story:** As a school IT administrator, I want a clear pre-event checklist for preparing student devices, so that all devices are ready for hackathon day with no surprises.

#### Acceptance Criteria

1. THE Setup_Guide SHALL present a numbered pre-event checklist covering: downloading and installing Kiro from kiro.dev/downloads, downloading and installing Node.js LTS, ensuring Google Chrome is installed, downloading the starter project repo ZIP file, and opening Kiro to sign in with Google and confirm it launches.
2. THE Setup_Guide SHALL include a "Verify it works" section with test steps: open Kiro, type a test prompt, and confirm a response is received.
3. THE Setup_Guide SHALL display a warning Admonition stating that Kiro does not run on ChromeOS and requires macOS, Windows (x64), or Linux.
4. THE Setup_Guide SHALL list minimum system requirements: macOS (Intel or Apple Silicon), Windows 64-bit (ARM not supported), and stable internet connection.
5. THE Setup_Guide SHALL state that Kiro requires internet access for AI interactions.
6. THE Setup_Guide SHALL include a placeholder for support contact details.
7. THE Setup_Guide SHALL note that Kiro is free and does not require an AWS account.
8. THE Setup_Guide SHALL be written in a professional, concise tone appropriate for IT administrators.

### Requirement 5: Student Guide

**User Story:** As a high school student with little to no coding experience, I want a friendly step-by-step guide for hackathon day, so that I can follow along and build my project without feeling lost.

#### Acceptance Criteria

1. THE Student_Guide SHALL present numbered steps covering: opening Kiro, opening the team project folder, reading the `requirements.md` file and finding TODO items, opening the Kiro chat and typing a first prompt, watching Kiro generate specs and code, previewing the app in Chrome, iterating by asking Kiro to add features or fix things, and stopping coding when the mentor announces "10 minutes left" to take screenshots for the pitch.
2. THE Student_Guide SHALL use Admonition callout boxes for tips and warnings throughout the guide.
3. THE Student_Guide SHALL include a link to the Troubleshooting_Page for when things go wrong.
4. THE Student_Guide SHALL be written in a friendly, encouraging tone with zero technical jargon.

### Requirement 6: Prompt Library

**User Story:** As a student, I want a library of ready-to-use prompts organised by problem theme, so that I can quickly get started and iterate on my project idea.

#### Acceptance Criteria

1. THE Prompt_Library SHALL organise prompts under five problem themes: Ocean/Water Pollution, Youth Wellbeing, Community Food Sharing, Native Species Conservation, and School Sustainability.
2. FOR EACH problem theme, THE Prompt_Library SHALL provide one starter prompt, two to three iteration prompts, and one stretch prompt, all formatted as copy-paste code blocks.
3. THE Prompt_Library SHALL include a section explaining what makes a good prompt compared to a vague prompt, with examples of each.

### Requirement 7: Troubleshooting Page

**User Story:** As a hackathon participant or mentor, I want a scannable list of common issues and fixes, so that I can resolve problems quickly without losing build time.

#### Acceptance Criteria

1. THE Troubleshooting_Page SHALL present issues in a problem-then-solution format that is easy to scan.
2. THE Troubleshooting_Page SHALL cover the following issues: macOS Gatekeeper blocking Kiro, Windows SmartScreen blocking Kiro, ARM device incompatibility, Kiro not responding, app preview not working in the browser, resetting broken code, Kiro running slowly, and inability to sign in.
3. WHERE an issue has platform-specific solutions, THE Troubleshooting_Page SHALL use Tab_Component to separate macOS and Windows instructions.

### Requirement 8: Mentor Guide

**User Story:** As an adult mentor or facilitator, I want a playbook that tells me exactly what to do and when during the hackathon, so that I can effectively support student teams.

#### Acceptance Criteria

1. THE Mentor_Guide SHALL include a role description explaining the mentor's responsibilities during the hackathon.
2. THE Mentor_Guide SHALL provide a 40-minute playbook with minute-by-minute guidance for the prototype/build phase.
3. THE Mentor_Guide SHALL include a first-5-minutes triage decision tree to help mentors assess and support teams at the start.
4. THE Mentor_Guide SHALL include prompt coaching tips for helping students write effective prompts.
5. THE Mentor_Guide SHALL include conversation starters for engaging with student teams.
6. THE Mentor_Guide SHALL describe what judges look for in the final presentations.

### Requirement 9: Resources Page

**User Story:** As a hackathon participant, mentor, or IT administrator, I want a single page with all relevant links and downloads, so that I can find external resources without searching.

#### Acceptance Criteria

1. THE Resources_Page SHALL list five themed starter repo links (one per problem theme) with placeholder URLs.
2. THE Resources_Page SHALL provide ZIP download links for schools without Git access, with placeholder URLs.
3. THE Resources_Page SHALL link to the pitch template, judging criteria document, Kiro documentation, Node.js downloads page, and event schedule, using placeholder URLs where final links are not yet available.

### Requirement 10: Content Formatting Standards

**User Story:** As a site visitor on any device, I want the hackathon pages to use consistent Docusaurus formatting and be mobile-friendly, so that the content is readable and easy to navigate.

#### Acceptance Criteria

1. THE Hackathon_Section SHALL use Docusaurus Admonition components for tips, warnings, notes, and important callouts across all pages.
2. WHERE instructions differ between macOS and Windows, THE Hackathon_Section SHALL use Tab_Component to present platform-specific content.
3. THE Hackathon_Section SHALL use fenced code blocks with copy buttons for all prompt examples and command-line instructions.
4. THE Hackathon_Section SHALL render correctly on mobile devices using the existing Docusaurus responsive layout without requiring custom CSS.
5. THE Hackathon_Section SHALL use only Markdown and MDX file formats for all content pages.
6. THE Hackathon_Section SHALL NOT introduce any external dependencies beyond those already present in the Docusaurus site.
